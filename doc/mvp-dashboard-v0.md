# Darktime MVP Dashboard v0

## Goal

Darktime is not a calendar client in this MVP. It is a local time control surface for agent access.

Dashboard v0 should make four things obvious:

- Which calendar sources Darktime can access.
- Whether local agent access is available.
- Which agents have recently called Darktime through MCP.
- What actions agents have attempted or completed.
- Whether the visible dashboard is fresh after an MCP process writes activity.

## Product Shape

The first screen should behave like a local control console:

```text
Darktime

Status
- Apple Calendar: connected / needs access / unavailable
- MCP: ready
- Last agent activity

Sources
- Apple Calendar: implemented, permission status, calendars, writable target
- Google Calendar: planned
- Feishu Calendar: planned
- Outlook: planned
- WeCom: planned

Agent Access
- MCP mode: stdio
- MCP command
- Storage path
- Recent sessions / last seen

Activity
- Recent reads, writes, blocks, and errors
```

## Important Boundary

Darktime currently exposes an MCP server over stdio.

That means there is no persistent localhost port yet. An MCP client such as Codex or Claude Code launches the MCP server process, and that server launches Darktime for authorized calendar operations.

Dashboard v0 should therefore show:

- `MCP mode: stdio`
- `Service health: ready` when the app bundle and SQLite store are available.
- No fake port number.

A future version can add a local daemon and then show port, uptime, and daemon health.

## SQLite Storage

Use a shared SQLite database at:

```text
~/Library/Application Support/Darktime/darktime.sqlite3
```

Override for development:

```text
DARKTIME_DB=/path/to/darktime.sqlite3
```

### Tables

`mcp_sessions`

- `id`: UUID generated by the MCP server process.
- `client_name`: currently `MCP stdio client` until the SDK exposes client info cleanly.
- `client_version`: optional.
- `transport`: `stdio`.
- `started_at`: ISO timestamp.
- `last_seen_at`: ISO timestamp.
- `last_tool_name`: last tool called.
- `last_tool_status`: `success`, `error`, or `blocked`.
- `tool_call_count`: number of tool calls in this session.

`action_logs`

- `id`: autoincrement integer.
- `created_at`: ISO timestamp.
- `session_id`: MCP session id.
- `client_name`: current client display name.
- `source`: `apple_calendar`, `mcp`, or another future source id.
- `action`: tool or system action name.
- `status`: `success`, `error`, `blocked`, or `started`.
- `is_write`: `1` for create/update/delete, otherwise `0`.
- `summary`: short human-readable summary.
- `error_code`: optional.
- `error_message`: optional.
- `request_json`: optional JSON payload.
- `response_json`: optional JSON result or error payload.

## Dashboard v0 Data Rules

Sources:

- Apple Calendar comes from EventKit authorization and visible calendars.
- Other providers are shown as planned or not implemented.
- The default write target should flag local-only calendars that may not sync to iPhone.

Agent Access:

- Show MCP command.
- Show DB path.
- Show recent MCP sessions sorted by last seen.

Activity:

- Show the latest action logs.
- Write actions must be visibly marked.
- Blocked writes caused by missing `confirm: true` should be logged.
- Refresh automatically while the app is open, because MCP stdio calls can happen after the dashboard has already launched.
- Prefer human-readable event details from `request_json` over raw truncated JSON summaries.

## Out Of Scope

- Full calendar UI.
- Google/Feishu/Outlook/WeCom OAuth.
- A persistent localhost daemon and port.
- Notarized public distribution.
- Fine-grained per-agent permission policy.

## Acceptance Criteria

- `npm run build:all` passes.
- `npm run build:dmg` passes.
- MCP tool calls create or update SQLite records.
- Darktime UI shows source status, MCP mode, DB path, recent sessions, and recent actions.
- Existing calendar read/write tools continue to work.
